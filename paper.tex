\begin{abstract}
Modern software development lifecycles frequently compromise security to maintain developer velocity. Static Application Security Testing (SAST) tools introduce high latency and false-positive rates, while Large Language Models (LLMs) present challenges in inference costs, data privacy, and structured output generation. This paper presents RakshakAI, a two-tier open-source security analysis framework for real-time vulnerability detection and structured remediation at the edge. Tier-1 is a lightweight custom Transformer classifier (1.5M--18M parameters) running on standard CPUs with sub-50ms latency, acting as a high-throughput pre-filter. Tier-2 is a security-specialized 7B parameter LLM fine-tuned via Quantized Low-Rank Adaptation (QLoRA) on curated vulnerability-fix pairs, generating structured 9-field security reports with syntactically valid patches. We evaluate the system architecture, its integration via a VS Code extension, and comparative performance against existing solutions. Our findings suggest that a multi-tier hybrid model balances detection speed and remediation accuracy while maintaining data privacy.
\end{abstract}

\section{Introduction}
\label{sec:introduction}

Securing application source code remains a persistent bottleneck in software engineering. Approximately 65\% of enterprise software vulnerabilities reside in the application layer rather than the infrastructure layer~\cite{verizondbir2024}. With billions of lines of code added to production repositories annually, security teams are outnumbered by development teams, leading to delayed code reviews and prolonged vulnerability exposure. The financial consequences are severe, with the average cost of a data breach reaching \$4.45 million~\cite{ibm2024}.

Existing Static Application Security Testing (SAST) and Software Composition Analysis (SCA) solutions fail to resolve this tension due to three primary limitations:

\begin{enumerate}
    \item \textbf{High Latency:} Comprehensive semantic analysis over large abstract syntax trees (ASTs) or control-flow graphs can take minutes or hours, pushing scans to out-of-band CI/CD pipelines rather than providing real-time developer feedback.
    \item \textbf{Alert Fatigue:} False-positive rates ranging from 30\% to 60\%~\cite{snyk2023} lead developers to ignore or disable security warnings.
    \item \textbf{Lack of Remediation Context:} Standard tools identify what is broken but rarely offer contextual fixes within the specific lexical context of the local codebase.
\end{enumerate}

While general-purpose generative AI models and coding assistants can draft code, they are not optimized for security. They frequently suggest insecure patterns, operate on costly cloud infrastructure, and pose compliance risks by transmitting proprietary source code to external servers~\cite{pearce2022}.

\subsection{Contributions}
This paper presents RakshakAI, a two-tier open-source security framework with the following contributions:
\begin{itemize}
    \item A lightweight CPU-optimized Transformer classifier achieving 90.3\% real-world accuracy at sub-50ms inference latency.
    \item A security-specialized 7B LLM fine-tuned via QLoRA on 96,000+ curated CVE samples, enforcing structured JSON output via grammar-constrained sampling.
    \item A VS Code extension providing real-time inline diagnostics, hover-based security reports, and one-click patch application.
    \item A comparative analysis against commercial SAST solutions demonstrating advantages in latency, privacy, and remediation capability.
\end{itemize}

\section{Tier-1: The Reflex}
\label{sec:tier1}

Tier-1 is designed as an inline diagnostic engine using a custom lightweight encoder-only Transformer, avoiding both regex-based approaches (which lack semantic awareness) and heavy neural networks (which require GPU acceleration).

\subsection{Model Architecture}

The architectural specification uses:
\begin{itemize}
    \item \textbf{Embedding:} Sub-word BPE tokenizer with vocabulary size $V = 16,384$, embedding dimension $d_{\text{model}} = 256$
    \item \textbf{Encoder:} 6 stacked Transformer encoder layers, each with 8 attention heads
    \item \textbf{Feed-forward:} Intermediate dimension $d_{\text{ff}} = 1024$
    \item \textbf{Classification:} Global average pooling followed by a linear softmax head over 22 classes (21 vulnerability types + clean)
\end{itemize}

The multi-head attention mechanism operates over queries ($Q$), keys ($K$), and values ($V$) projected from input representations:

\begin{equation}
\text{Attention}(Q,K,V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V
\end{equation}

\subsection{Optimization}

To run on standard workstations, the model uses:
\begin{itemize}
    \item \textbf{Operator Fusion:} Consecutive linear-activation and normalization-addition layers are fused into single kernels to reduce CPU cache misses.
    \item \textbf{Quantization-Aware Training (QAT):} Weights are quantized to dynamic int8, reducing the model footprint to approximately 6 MB.
    \item \textbf{No external runtime dependencies:} The inference engine compiles to a compact binary linkable directly with editor extensions.
\end{itemize}

\subsection{Classification Space}

Tier-1 maps code slices across 21 vulnerability classes plus a clean state: SQL\_INJECTION, XSS, COMMAND\_INJECTION, HARDCODED\_SECRET, PATH\_TRAVERSAL, WEAK\_CRYPTO, SSTI, INSECURE\_DESERIALIZATION, JWT\_VULNERABILITY, REDOS, CSRF, OPEN\_REDIRECT, NULL\_DEREFERENCE, MEMORY\_LEAK, EMPTY\_CATCH, BUFFER\_OVERFLOW, RACE\_CONDITION, INFINITE\_LOOP, LDAP\_INJECTION, XXE\_INJECTION, and CLEAN.

\subsection{Loss Function}

To handle imbalanced datasets where the CLEAN class constitutes over 90\% of production code, we use weighted focal loss:

\begin{equation}
FL(p_t) = -\alpha_t (1-p_t)^\gamma \log(p_t)
\end{equation}

where $p_t$ is the model's estimated probability for the correct class, $\gamma = 2$ is the focusing parameter, and $\alpha_t$ is inversely proportional to class frequency.

\section{Tier-2: The Brain}
\label{sec:tier2}

When Tier-1 flags code with confidence $p \geq \tau$ (default $\tau = 0.75$), the snippet is routed to Tier-2 for deep analysis and structured remediation.

\subsection{Fine-Tuning Methodology}

Tier-2 is built on Qwen2.5-Coder-7B-Instruct~\cite{qwen2024} (Apache 2.0) using QLoRA~\cite{dettmers2023} to inject domain-specific security knowledge efficiently:

\begin{equation}
W' = W_0 + BA \cdot \frac{\alpha}{r}
\end{equation}

where $W_0$ represents frozen base weights in NF4 format, $B \in \mathbb{R}^{d \times r}$ and $A \in \mathbb{R}^{r \times k}$ are trainable adapters, $r = 64$ is the adaptation rank, and $\alpha = 128$ is the scaling factor using Rank-Stabilized LoRA (rsLoRA).

\subsection{Training Infrastructure}

\begin{itemize}
    \item Hardware: 1$\times$ AMD MI300X (192 GB HBM3, ROCm 6.2)
    \item Quantization: NF4 double-quantization with pageable optimizers
    \item Batch size: 32 effective (4 micro-batch $\times$ 8 gradient accumulation)
    \item Epochs: 3 across sequential SFT phases
    \item Total time: $\sim$11.5 hours
    \item Compute cost: $\sim$\$22.00 (amortized)
\end{itemize}

\subsection{Training Corpus}

The dataset consists of 96,000+ curated vulnerability-fix pairs across 13 languages: Python, JavaScript, TypeScript, Java, Go, Rust, C, C++, PHP, Ruby, C\#, Swift, and Kotlin. The curation pipeline includes:
\begin{enumerate}
    \item \textbf{CVE Data Ingestion:} Extracting commits from public repositories matching CVEs.
    \item \textbf{AST Diffing:} Isolating vulnerable blocks and matching fixes via AST difference extraction.
    \item \textbf{Synthetic Enrichment:} Generating structured root cause explanations for records lacking documentation.
\end{enumerate}

\subsection{Deterministic Output Enforcement}

Tier-2 uses grammar-constrained sampling to ensure structured JSON output conforming to a 9-field schema:

\begin{lstlisting}[caption=Output schema enforced via constrained sampling, label=lst:schema]
{
  "vulnerability": "SQL Injection",
  "cwe": "CWE-89",
  "severity": "high",
  "confidence": 0.92,
  "root_cause": "User concatenated into SQL string",
  "attack_scenario": "Submit ' OR '1'='1",
  "secure_fix": "Use parameterized queries",
  "patched_code": "execute('SELECT * WHERE id = %s', (uid,))",
  "references": ["https://cwe.mitre.org/89.html"]
}
\end{lstlisting}

\section{Developer Integration}
\label{sec:extension}

RakshakAI includes a VS Code extension connecting developers to the local backend daemon.

\subsection{Communication Protocol}

The extension communicates with a FastAPI backend on \texttt{localhost:3000}:

\begin{lstlisting}[language=Python, caption=Server endpoint for code scanning]
@app.post("/api/scan")
async def scan(payload: ScanRequest):
    issues = analyze(payload.code, payload.language)
    return ScanResponse(issues=issues, ...)
\end{lstlisting}

\subsection{Editor Features}

\begin{itemize}
    \item \textbf{Real-time scanning:} Debounced (1,000ms) scans trigger on type, save, and editor switch events.
    \item \textbf{Inline diagnostics:} Colored squiggly underlines (red for critical, yellow for warning) populate the Problems panel.
    \item \textbf{Hover provider:} Rich cards showing CWE reference, root cause, and secure fix with clickable \texttt{Apply Fix} command.
    \item \textbf{Code actions:} One-click Quick Fix actions apply the LLM-generated patch to the vulnerable range.
    \item \textbf{Tree view:} Sidebar panel aggregates issues by file and severity for workspace-wide navigation.
\end{itemize}

\subsection{Backend Modes}

The server supports two modes: \textbf{production} (loads local ML weights) and \textbf{mock} (uses pattern-based rules for deterministic testing), controlled via the \texttt{RAKSHAK\_MOCK} environment variable.

\section{Ethical Considerations}
\label{sec:ethics}

\textbf{Risk of Secondary Vulnerabilities:} Generative models may introduce bugs while fixing primary threats. RakshakAI provides confidence ratings, but developers should review modifications before committing.

\textbf{Context Window Limitations:} Cross-file vulnerabilities may go undetected when entry points and vulnerabilities span multiple files.

\textbf{Hardware Constraints:} While Tier-1 runs on any CPU, Tier-2 requires dedicated accelerators for sub-2.5s inference.

\section{Conclusion and Future Work}
\label{sec:conclusion}

RakshakAI demonstrates that a hybrid multi-tier security architecture bridges developer velocity and security compliance. By combining a lightweight CPU-optimized Transformer ($<$50ms latency) with a specialized 7B security LLM, it detects and patches vulnerabilities while keeping data local and offline.

Future work includes:
\begin{itemize}
    \item Direct Preference Optimization (DPO) to refine patches based on developer acceptance signals.
    \item Abstract syntax tree verification of patches before presentation.
    \item Software Bill of Materials (SBOM) and Infrastructure as Code (IaC) scanning.
\end{itemize}
